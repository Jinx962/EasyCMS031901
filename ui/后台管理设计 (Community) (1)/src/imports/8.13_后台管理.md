

# 8.13 后台管理

## 8.13.1 模块定位

后台管理是地面内容管理平台的基础支撑模块，负责管理平台内的**用户、角色、权限、通知策略和操作审计能力**，确保不同岗位用户在合适的页面、合适的操作范围内完成工作，并保证关键业务动作可追踪、可审计、可回溯。

后台管理不是独立业务系统，而是贯穿内容中心、周期管理、配置中心、导出发布、报表中心等模块的统一治理层。系统管理员通过后台管理控制“谁能看、谁能改、谁能批、谁会收到通知、谁做过什么”。这与现有 PRD 中“系统管理员负责用户、角色、权限、后台参数与系统运维能力管理”“平台需支持用户管理、角色管理、页面级权限、操作级权限、只读权限，并保留内容编辑、签核、变更审批、配置修改、导入、导出等审计日志”的定义一致。


---

## 8.13.2 模块目标

|编号|目标|说明|
|:----|:----|:----|
|8.13.2.1|用户治理|统一维护平台用户账号、状态、所属组织与岗位信息|
|8.13.2.2|角色治理|通过角色模板快速定义不同岗位的访问范围与操作边界|
|8.13.2.3|权限控制|支持页面级、功能级、操作级权限控制，并支持只读模式|
|8.13.2.4|审计留痕|对关键业务动作形成完整日志，满足追责与排障要求|
|8.13.2.5|通知触达|对导出、签核、解锁、变更、任务等事件进行消息通知|
|8.13.2.6|安全隔离|避免越权访问、误操作和非授权变更|
|8.13.2.7|运维提效|降低账号开通、权限调整、问题排查的人力成本|


---

## 8.13.3 模块边界

### 8.13.3.1 本模块负责

1. 用户账号管理

2. 用户状态管理

3. 用户与角色绑定

4. 角色定义与复制

5. 页面级权限配置

6. 操作级权限配置

7. 只读权限策略

8. 审计日志查询与导出

9. 通知规则与通知接收对象配置

10. 系统级后台参数（首期如不做可保留占位）

### 8.13.3.2 本模块不负责

1. 业务内容本身的编辑

2. 周期、配置、导出的业务执行

3. 站内消息中心的完整 IM 能力

4. 第三方身份认证平台建设

5. 完整的安全中心能力（如风控、设备指纹、异常登录分析）

## 8.13.4 产品结构

后台管理建议拆为 5 个二级模块：

|一级模块|二级模块|说明|
|:----|:----|:----|
|后台管理|用户管理|管理用户账号、状态、组织、角色绑定|
|后台管理|角色与权限|管理角色定义、页面权限、操作权限、只读权限|
|后台管理|审计日志|查询内容、签核、变更、配置、导入、导出等日志|
|后台管理|通知中心|管理通知事件、通知方式、接收人范围|
|后台管理|系统参数|管理基础系统参数、默认策略、枚举配置（可选）|

### 推荐左侧菜单

* 用户管理

* 角色管理

* 权限配置

* 审计日志

* 通知配置

* 系统参数（可选）

其中，“角色管理”和“权限配置”可合并为一个模块，也可拆开。

如果首期强调效率，建议拆开，避免角色列表页过于复杂。

# 8.13.7 用户管理

## 8.13.7.1 功能目标

统一维护平台用户账号及其组织、角色、状态、通知接收范围等信息，支撑账号全生命周期管理。

## 8.13.7.2 页面结构

### A. 用户管理列表页

* 关键字搜索（用户名/姓名/邮箱/手机号）

* 筛选项：组织、角色、账号状态、创建时间、最后登录时间

* 按钮：新建用户、批量启用、批量停用、导出列表

中部表格区：

* 用户名

* 姓名

* 邮箱

* 手机号

* 所属组织

* 角色数

* 账号状态

* 最近登录时间

* 创建时间

* 操作

右上角支持列设置。

### B. 用户详情页

Tab 建议：

* 基本信息

* 角色信息

* 权限概览

* 通知设置

* 操作历史

|8.13.7.5.3|启用/停用用户|控制账号可用性|P0|

## 8.13.7.6 页面流程

### 场景 1：新建用户

用户管理列表 → 点击“新建用户” → 填写基础资料 → 选择组织 → 选择角色 → 设置账号状态为“未激活/启用” → 保存成功 → 系统写入审计日志 → 按通知策略发送激活通知

### 场景 2：调整用户角色

用户管理列表 → 打开用户详情 → 进入角色信息 Tab → 增减角色 → 系统校验是否有冲突角色 → 保存 → 更新最终权限 → 写入审计日志 → 可选发送角色变更通知

### 场景 3：停用离岗人员

搜索用户 → 点击“停用” → 二次确认 → 系统立即禁止登录 → 保留历史日志和责任追踪关系 → 写入审计日志

## 8.13.7.7 规则说明

1. 用户名不可重复

2. 已停用用户不可登录，但历史操作记录需保留

3. 用户可以绑定多个角色，最终权限按“并集”计算，冲突时以“禁止优先”

4. 至少保留一个超级管理员账号不可被停用

5. 同一用户在多个项目中的可见范围由“所属项目 + 角色范围”共同决定

6. 账号停用不影响已产生的业务记录归属

* 内容服务商

* 发布人员

* 配置管理员

* 只读用户

* 支持排障人员

这与现有 PRD 的角色边界基本一致：系统管理员负责账号与权限，业务角色分别负责内容、周期、审核、配置和只读查看。

## 8.13.8.6 规则说明

3. 停用角色后，不影响已绑定用户的历史日志，但不允许继续分配

4. 角色权限调整后，对绑定用户实时生效或在下次登录生效，需由技术方案确定

5. 角色必须至少绑定一个页面权限，否则不可启用

# 8.13.9 权限配置

## 8.13.9.1 功能目标

控制不同角色在平台中的**访问范围、可操作动作和数据边界**。

现有 PRD 已明确后台管理需支持**页面级权限、操作级权限、只读权限**，并要求不同角色只能访问授权模块和操作，且只读角色不得修改任何内容与配置。

## 8.13.9.2 权限模型

建议采用四层模型：

1. 模块权限

2. 页面权限

3. 操作权限

4. 数据范围权限

### 1）模块权限

控制是否可见一级菜单，如：

* 工作台

* 内容中心

* 周期管理

* 报表中心

* 配置中心

* 后台管理

* 帮助中心

### 2）页面权限

控制某模块下的具体页面是否可访问，如：

* 内容详情页

* 状态清单页

* 周期详情页

* 导出列表页

* 配置详情页

* 用户管理页

* 审计日志页

### 3）操作权限

控制具体按钮或动作，如：

* 新建

* 编辑

* 删除

* 导入

* 导出

* 签核

* 驳回

* 解锁

* 变更审批

* 启用/停用

* 下载

* 查看日志

### 4）数据范围权限

控制能看到哪些数据，如：

* 全部项目

* 所属组织

* 所属项目

* 本人创建

* 指定项目集

## 8.13.9.3 权限配置页面结构

建议采用“左树右矩阵”布局。

左侧：

* 一级模块树

* 二级页面树

右侧：

* 页面可见

* 查看

* 新建

* 编辑

* 删除

* 导入

* 导出

* 审批

* 解锁

* 日志查看

* 其他扩展动作

底部：

* 保存

* 重置

* 复制自其他角色

* 权限预览

## 8.13.9.4 只读权限设计

### 只读模式定义

只读不是单独页面，而是一组权限策略：

* 可访问页面

* 可查看详情

* 不允许编辑

* 不允许导入

* 不允许导出发起

* 不允许审批

* 不允许配置修改

* 不允许删除

### 只读用户推荐访问范围

根据你现有 PRD 的描述，只读用户用于排查问题、查看配置、查看内容与导出结果，不允许修改，因此建议默认开放：

* 工作台

* 状态清单

* 内容中心详情查看

* 周期管理查看

* 报表中心查看

* 配置中心查看

* 帮助中心不开放：

* 后台管理修改

* 导入执行

* 导出发起

* 配置编辑

* 审批动作

## 8.13.9.5 权限配置规则

|编号|规则|
|:----|:----|
|8.13.9.5.1|权限变更仅系统管理员可操作|
|8.13.9.5.2|未授权页面在导航中隐藏，而非展示后报无权限|
|8.13.9.5.3|用户直接访问无权限页面时，返回“无访问权限”页|
|8.13.9.5.4|操作按钮按权限显示，不允许前端仅置灰但后端可调用|
|8.13.9.5.5|后端必须再次校验权限，避免越权调用|
|8.13.9.5.6|若同时拥有允许和禁止权限，以禁止优先|
|8.13.9.5.7|只读权限不可绕过操作级权限直接修改数据|

## 8.13.9.6 典型权限场景

### 场景 A：内容服务商

可访问：工作台、内容中心、导入历史、状态清单

可操作：导入、编辑被分配内容、上传素材

不可操作：签核审批、配置编辑、导出发布、后台管理

### 场景 B：审核人员

可访问：工作台、内容中心、状态清单、报表中心

可操作：初始签核、全量签核、变更审批

不可操作：用户管理、角色管理、系统参数

### 场景 C：发布人员

可访问：周期管理、导出发布、报表中心

可操作：完整性校验、创建导出任务、查看导出日志

不可操作：内容底层元数据编辑、后台权限变更

### 场景 D：只读排障人员

可访问：大部分查询页面

可操作：仅查看、下载报表

不可操作：新建、编辑、删除、审批、导入、导出发起

# 8.13.10 审计日志

## 8.13.10.1 功能目标

对平台关键操作进行全链路留痕，用于问题排查、责任认定、变更追溯和合规审计。

你现有 PRD 已经明确需要记录：**内容编辑日志、签核日志、变更审批日志、配置修改日志、导入日志、导出日志**，且“所有关键操作必须记录时间、操作者、对象和结果”。

## 8.13.10.2 日志分类

|编号|日志类型|说明|
|:----|:----|:----|
|8.13.10.2.1|内容编辑日志|记录内容字段、图片、字幕、音轨等编辑行为|
|8.13.10.2.2|签核日志|记录初始签核、全量签核、解锁等行为|
|8.13.10.2.3|变更审批日志|记录变更单提交、审批、驳回、取消|
|8.13.10.2.4|配置修改日志|记录配置档、分类集、媒体配置、路由等修改|
|8.13.10.2.5|导入日志|记录导入对象、模式、结果、错误明细|

## 8.13.10.5 日志记录规则

1. 所有写操作必须记录日志

2. 审批类动作必须记录操作者、意见和结果

3. 导入导出类任务必须记录任务ID

4. 失败操作也必须记录，便于排查

5. 日志默认不可编辑、不可删除

6. 普通管理员仅可查看授权范围内日志，系统管理员可查看全部

7. 日志应支持按对象反查，例如“某内容最近 30 天被谁改过”

# 8.13.11 通知中心

## 8.13.11.1 功能目标

将平台关键业务事件及时通知到相关角色或指定用户，避免导出失败、审批延迟或配置风险无人感知。

现有 PRD 已明确平台需支持：**导出结果通知、配置签核通知、配置解锁通知、变更申请通知、任务提醒**。

## 8.13.11.2 通知事件范围

|编号|通知事件|触发时机|
|:----|:----|:----|
|8.13.11.2.1|导出结果通知|导出完成、失败、存在严重错误|
|8.13.11.2.2|配置签核通知|配置被签核或确认生效|
|8.13.11.2.3|配置解锁通知|配置被解锁、允许再次编辑|
|8.13.11.2.4|变更申请通知|提交、审批通过、驳回、取消|
|8.13.11.2.5|任务提醒|到期、逾期、待处理任务提醒|
|8.13.11.2.6|账号通知|新账号开通、角色变更、账号停用|
|8.13.11.2.7|高风险操作提醒|例如批量停用、配置覆盖导入、重复导出被拦截|

## 8.13.11.3 通知方式

* 站内消息

* 邮件

* 短信（可选）

* Webhook/外部消息通道（后续）

## 8.13.11.4 通知配置页结构

## 8.13.11.5 接收对象规则

支持以下接收方式：

1. 固定用户

2. 固定角色

3. 事件相关责任人

4. 对象创建人/审批人/导出发起人

5. 组织管理员

6. 项目管理员

## 8.13.11.6 通知规则说明

1. 同一事件可配置多个通知方式

2. 高并发事件支持合并通知，避免消息轰炸

3. 通知失败需进入通知日志

4. 高风险动作默认通知管理员

5. 用户可配置个人通知接收偏好，但不得关闭强制通知

## 8.13.11.7 典型场景

### 场景 1：导出失败通知

发布人员发起导出 → 校验失败 → 系统发送站内消息 + 邮件 → 通知发布人员、项目管理员、相关审核人

### 场景 2：变更申请待审批

内容运营提交变更申请 → 系统通知审核人员 → 若 24 小时未处理再次提醒 → 超时通知主管

### 场景 3：配置解锁通知

配置管理员提交解锁 → 审批通过 → 系统通知配置管理员和发布人员，提示当前配置进入可编辑状态

# 8.13.13 关键页面流程

## 8.13.13.1 用户开通流程

系统管理员进入用户管理 → 新建用户 → 填写基础信息 → 绑定组织与角色 → 保存 → 系统写入用户创建日志 → 发送激活通知 → 用户首次登录激活

## 8.13.13.2 角色配置流程

系统管理员进入角色管理 → 新建/复制角色 → 填写角色名称与说明 → 进入权限配置页 → 勾选页面权限、操作权限和数据范围 → 保存 → 系统写入角色变更日志

## 8.13.13.3 权限排查流程

支持人员进入审计日志 → 按对象或用户查询 → 查看某次失败操作或异常访问 → 如确认权限缺失 → 系统管理员进入角色详情 → 调整权限 → 保存 → 写入审计日志

## 8.13.13.4 通知排查流程

用户反馈“未收到导出通知” → 系统管理员进入通知中心 → 查看通知日志 → 检查事件是否启用、接收对象是否命中、发送是否成功 → 若失败则展示失败原因并支持重发

# 8.13.14 典型用户场景

## 场景一：新项目启动，快速开通一批账号

**角色**：系统管理员

**目标**：为内容运营、审核、发布、配置管理员开通账号

**路径**：后台管理 → 用户管理 → 新建用户/批量导入 → 分配角色 → 启用账号

**价值**：快速完成项目初始权限配置

## 场景二：只读排障人员需要查看问题但不能修改

**角色**：系统管理员、排障人员

**目标**：给支持人员开放查询能力，但禁止修改业务数据

**路径**：后台管理 → 角色管理 → 创建“只读排障”角色 → 配置只读权限 → 绑定用户

**价值**：兼顾排障效率与业务安全

## 场景三：发生误改，需要追踪是谁修改了配置

**角色**：配置管理员、系统管理员

**目标**：追踪某配置被谁在什么时间修改

**路径**：后台管理 → 审计日志 → 筛选“配置修改日志” → 查询对象 → 查看前后值

**价值**：降低扯皮，快速定位责任和恢复依据

## 场景四：导出失败后自动通知相关负责人

**角色**：发布人员、审核人员、项目管理员

**目标**：导出失败后自动拉起处理

**路径**：后台管理 → 通知配置 → 配置导出失败通知规则

**价值**：降低因消息缺失导致的问题滞后

## 场景五：离岗人员账号停用

**角色**：系统管理员

**目标**：停用账号，避免继续登录或误操作

**路径**：后台管理 → 用户管理 → 搜索用户 → 停用

**价值**：保证账号安全和权限回收

# 8.13.15 交互规则与异常处理

## 8.13.15.1 通用交互规则

1. 列表页统一支持搜索、筛选、排序、分页

2. 高风险操作必须二次确认

3. 删除类操作需说明影响范围

4. 权限不足时按钮不展示或置灰并提示原因

5. 保存成功需展示 toast，失败需给出明确原因

6. 审计日志详情支持字段级 diff 展示

## 8.13.15.2 异常处理

|异常场景|处理方式|
|:----|:----|
|用户名重复|阻止保存，提示用户名已存在|
|角色名称重复|阻止保存，提示角色已存在|
|停用最后一个超级管理员|阻止操作|
|删除已绑定用户的角色|阻止删除并提示先解绑|
|配置通知规则无接收人|阻止启用|
|查询日志无结果|展示空状态和建议筛选条件|
|无权限访问后台管理|返回无权限页，不展示敏感信息|
|通知发送失败|记录通知日志，支持重试|
